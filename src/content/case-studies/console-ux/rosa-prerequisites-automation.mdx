Enabling ROSA used to mean verifying AWS account prerequisites by hand across several consoles, and the getting-started documentation customers expected on docs.aws.amazon.com did not exist. For the Q4 2022 console launch, I created the UI text and help panels for an automated prerequisite-validation experience and wrote the AWS-native getting-started documentation, designed together as one Day 1 experience. This was the foundation the later [ROSA with Hosted Control Planes Launch](/case-studies/console-ux/rosa-hcp-console-integration/) built on.

## The challenge

Before creating a cluster, customers had to confirm their AWS account met prerequisites for Service Quotas, IAM roles, and the ELB service-linked role. The check was entirely manual: discover the requirements in documentation, then verify them across multiple AWS consoles, with no proactive signal of readiness, so failures surfaced late after customers had already invested time.

The gap was wider than automation. AWS had no native getting-started documentation for ROSA; all Day 0 and Day 1 content lived on Red Hat's site. Customers who expected prerequisite and deployment information in the AWS service guide on docs.aws.amazon.com had to look elsewhere, which made ROSA feel less AWS-native than they expected.

## Automating prerequisite validation

The prerequisite page checks three account requirements automatically: ROSA enablement status, Service Quotas sufficiency, and the ELB service-linked role. I wrote the UI text and help panels that explain each check and show readiness in real time, including introductory text that sets expectations, real-time status with success and error states, actionable error messaging that identifies the specific quota to adjust, info callouts explaining why each prerequisite matters, and a validation checkpoint before customers proceed.

I created six help panels for customers coming from on-premises OpenShift who were new to AWS service integrations: ROSA Enablement, Verify Prerequisites, Service Quotas, ELB Service-Linked Role, AWS Account Requirements, and AWS and Red Hat Account Connection. Each linked to deeper documentation, creating a path from in-console validation to detailed configuration guidance.

## Closing the AWS-native documentation gap

Alongside the console work, I wrote the getting-started documentation for the AWS ROSA User Guide: a landing page routing customers to the right workflow (CLI-based or PrivateLink), a set-up page gathering all prerequisites in one place, and step-by-step deployment procedures. The prerequisites in the docs matched the in-console checks exactly, so customers got one consistent story whether they started in the console or the documentation. I worked with Red Hat's writers so both sites used consistent AWS service terminology and customers did not hit conflicting information across platforms.

## What this demonstrates

- Designing an automated console experience and its documentation as one Day 1 path
- Aligning in-console validation with documentation so the two never conflict
- Meeting customers where they expect to find information, on AWS-native docs
- Cross-company collaboration on terminology consistency

## Outcomes

The coordinated console and documentation experience shipped in Q4 2022, creating ROSA's first complete AWS-native enablement path and closing the gap customers hit when prerequisite information lived only on a partner site. It became the foundation for later ROSA work, including the 2023 HCP billing integration, and the patterns for automated prerequisite checking and coordinated console-documentation development became the template for future ROSA console work.
