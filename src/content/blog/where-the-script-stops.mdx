A few years ago I inherited a backlog of AWS service documentation locked in a legacy XML format, with a mandate to move it to AsciiDoc. The work began on ROSA, where we wrote alongside Red Hat's technical writers, and the XML toolchain was the wall between those writers and the docs they wanted to edit. The pile was thousands of files deep. My first instinct was the one I'd been trained into: open the queue and start converting.

If you've shipped documentation at any real size, you know that instinct, and you know where it ends. Convert by hand and you finish sometime next year, with a thousand small inconsistencies you introduced along the way. The job was bigger than hands. So I stopped converting and started building the thing that would convert. That shift, from writing the docs to engineering the system that produces them, is the part of the craft I want to talk about.

## The pipeline does three kinds of work

The first layer is deterministic. I wrote a Python toolkit that handled the mechanical part of the transform: parse the XML tree, map elements to their AsciiDoc equivalents, rewrite cross-references, move assets, preserve the structure that already had a clean one-to-one translation. This is the work a regex is good at and a person is bad at. It runs the same way on file one and file three thousand, and it doesn't get bored on file two thousand and start cutting corners.

Then the script stops. It stops at the files where the source encodes something the schema never named: a table that's really a definition list, an admonition an old author signaled with bold text and a shared convention instead of a tag, a sectioning choice that only makes sense if you know what the page is for. A deterministic rule can't rule on those, because the rule is the thing in question. That's where I pointed Claude agents. I handed them the conventions I'd settled on and let them make the per-file calls at a volume I couldn't reach by hand: read this messy source, work out what it was trying to be, render it in AsciiDoc the way the house style wants it.

The agents had a stopping point too, and it sat one level up, at the conventions themselves. What should the house style want when two valid renderings disagree? Which one is correct for this doc set, this audience, this product? Those questions don't have a deterministic answer or an agentic one. The agents flagged the cases they couldn't resolve, and the calls came back to me. I set the rules by hand and worked through the exceptions one at a time. The pipeline did the volume. I owned what the volume should look like.

(I wrote up the migration at the program level as a [case study](/case-studies/systems-strategy/adoc-modernization/) if you want the longer version of how the rollout actually went.)

## The seam can move

I'm building the same shape again now, larger, on a set of cross-product solution guides at Cloudflare. The transform there is a different one: turning scattered single-product docs into guides that walk a reader across several products to finish one job. The deterministic and agentic layers carry more here. OpenCode orchestrates multi-step Claude agents that draft, cross-check against the source docs, and assemble.

What's different is where my judgment lives. On the AWS migration I applied it case by case, ruling on exceptions as they surfaced. On the Cloudflare guides I encoded it into the pipeline as validation gates, the checks a draft has to pass before it's allowed to move downstream. The judgment moved out of the artifact and into the machine. The seam between human work and automated work isn't fixed in one place. You get to choose where it falls, and that choice is itself part of the design.

## What I take from both

The skill now is assigning each decision to the layer that should own it, a script, an agent, or me, and designing the handoffs so each one does the part it's genuinely good at. Draw the seams wrong and you either hand a machine a judgment call it will get confidently wrong, or you spend your afternoon doing by hand what a script should have finished before lunch. Draw them right and the documentation keeps pace with the engineering instead of falling behind it. If you've drawn the seams in your own pipeline differently, I'd genuinely like to read how.

This post is a small version of the same thing. I didn't write it cold. The conventions live in a few writing skills I maintain: a negative style guide, a set of voice rules, a structural framework. A Claude agent took those and ran the drafting flow, proposing a title, an outline, and a first draft that already followed the rules. I made the calls on what the post argues, which three projects anchor it, and which sentences survived the edit. The agent handled the production. The standards it worked against were mine, and so were the calls it couldn't make.

The craft didn't disappear when the typing got automated. It moved to the seams.
