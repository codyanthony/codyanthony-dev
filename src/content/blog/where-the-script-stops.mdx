I once ran documentation migrations from XML to AsciiDoc. Most of it was mechanical, the kind of structural conversion a script does the same way every time. The rest needed someone to read intent. A table might really be a definition list, for example. Or a section might make sense only if you knew why it had been built that way. Those calls were left to me.

That gap is where documentation systems tend to fail, not in the obvious breakages like broken builds or missing files, but in the quiet places where one kind of work stops being enough and another has to take over.

## The same problem, harder

The pipeline I built next produces documentation. It reads the platform's existing docs and drafts a guide that walks a reader across several products to finish one task. The migration had an original to check against. A guide has no single original. It assembles a path from sources that were never written as a single path, and the connective material has nothing to check it against. A reader acts on all of it.

## How far the scripts and agents get

The first layer is deterministic Python, the same role the XML mapping played in the migration. It finds candidate pages, scores each one for relevance on a fixed scale, reads git history to sort fresh content from stale, and checks every link. It returns the same output on run one and run fifty, and when it reports a broken link, that count is ground truth. This is the work a script is good at and a person is bad at. It's exhaustive, identical every time, no boredom on page two hundred. It also stops at a clean line. Ask it whether a page is *worth* reusing, or whether a paragraph frames a feature around the customer's goal or the product's name, and it has nothing to offer.

The second layer picks up there. I gave a set of Claude agents fixed rubrics, each a short menu of allowed answers. The page-health rubric has four options, good, needs-update, stale, or deprecated. The agent picks one and cites the line that justifies it, and it can't invent a fifth or write a sentence of its own. Other rubrics weigh how much can be reused and whether a page frames a feature around the customer's goal or the product. A second agent challenges the high-stakes calls before they're trusted.

I learned where that edge was by getting it wrong first. Early versions let the agents write into a free-form notes field, and most of what came back was confident, fluent, and too vague to act on. The accuracy jumped the moment I took the open field away.

## Clean, cited, and still wrong

The first time it caught me off guard, a draft was walking a reader through an interface the agent had never seen. It read perfectly. Every step cited a source page, the procedure flowed, nothing tripped a check. So I tried to follow it myself, and that's where it came apart. A checkbox in the source doc had become a toggle. An on-page label had become a tab. With no access to the live product, the agent had reconstructed the screens from the words on the page, the way you'd sketch a room from a description, and the sketch was confidently wrong. The rubric never stood a chance. It was checking the draft against the docs, and against the docs the draft was right.

The fix was to teach the system its own blind spot. Source docs became the ground truth, and anything the agent can't confirm against one now gets flagged for a human to check before it goes any further. That rule is obvious once you've been burned by it. But the system couldn't have proposed it on its own. The gap was invisible from inside the docs it was built to trust.

The second catch had nothing to do with accuracy and everything to do with who the guide was for. These guides solve problems that hit most of the platform's customers, and most of those customers are on the free plan. Left to its own devices, the pipeline kept reaching for the most capable way to solve a problem, and the core workflow quietly filled with features only paid tiers could touch. I didn't see it until I read one of the guides the way a free-plan user would, top to bottom, and hit a wall three steps in. The system had no concept that the solution should reach the same people who have the problem. So I went through the product tier by tier, found where the free plan runs out, and rebuilt the spine of every guide so a free-plan reader can finish it, with the paid features offered along the way as options. That call isn't in any rubric. It's a stance about who the work serves.

## When the architecture is the bug

Both of those are catches inside a system that ran exactly as built and handed me something I had to override. The harder case is when the machinery itself is built wrong, because then there's nothing to override. There's only a result that feels off, and no gate that can tell you why.

I ran into this with the draft pass. The pipeline runs on OpenCode, with Claude doing the reading and the writing and me steering. The audit, the pass that reads and assesses the docs, fans out a separate agent on each page at once. That's fast and loses nothing, because reading is independent work. I'd built the draft, the pass that writes the guide, the same way, a separate agent on each section at once. The result was wrong in a way I couldn't put my finger on at first. Sections contradicted each other. The same point showed up twice in different words. The voice drifted between paragraphs. It took several rounds of digging, with the best model I had at the time, before the cause came clear. A subagent doesn't carry the main context, so each parallel writer had lost the style rules, the structure rules, and any sense of what its neighbors were saying. Drafting isn't independent the way reading is. A guide is a single argument, and I'd split it among writers who couldn't see each other.

The fix was to stop parallelizing the draft and let one agent write the whole thing with the full context in scope. It runs slower. It's correct. No gate could have flagged this for me, because the flaw wasn't in any single output a gate could check. It was in the arrangement of the parts. The same pattern was right for the audit and wrong for the draft, and the only way to see that was to step back from the pipeline and look at its shape, which is not something a pipeline can do for you.

## The judgment that never stops

Every one of those catches went back into the system as a new check. Each one closed the gap it came from, and the next guide opened a new one the checks weren't built to see.

Here's the part I keep coming back to. The catching never drops to zero. The script stops where mechanical certainty stops; the agents stop at the edge of the decision space I drew; judgment is the layer with no defined end. It isn't a fallback that kicks in when the automation fails. People call it human in the loop. What the phrase means, once you've built the loop and watched it run, is the part that stays where the system can't see, asking the one question no rubric can answer: is the output what the reader on the other end actually needed? When the answer is no, it reshapes the system.
